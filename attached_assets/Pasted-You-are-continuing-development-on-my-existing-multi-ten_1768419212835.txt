FEATURE B: BULK TASK SECTIONS + TASKS (CSV IMPORT) FOR A PROJECT
===============================================================================
B1) UI Placement
In Tenant Drawer > Projects tab, when viewing a project row:
- Add actions:
  - “Seed Tasks”
  - “Bulk Import Tasks (CSV)”
Opens an inline panel (not a modal) within the drawer:
- Project selector (if not already in context)
- Section builder/import tool

B2) CSV Format (tasks)
Required columns:
- projectName OR projectId (choose one; prefer projectId if available in UI)
- sectionName
- taskTitle
Optional columns:
- description
- status
- priority
- dueDate (YYYY-MM-DD)
- assigneeEmail (single) OR assigneeEmails (comma-separated if multi-assign exists)
- tags (comma-separated)
- parentTaskTitle (for subtasks linking by title; optional)
- isSubtask (true/false) OR infer if parentTaskTitle present

Rules:
- Sections created if they do not exist for that project (safe).
- Tasks created under section.
- Subtasks:
  - Only one level deep.
  - If parentTaskTitle references a task created in the same import batch, link it.
  - If parent task not found, mark row error (do not create orphan subtasks).
- Validate status/priority against existing enums; if invalid => error row.
- If assigneeEmail not found => error row unless “create placeholder users” option is explicitly enabled (default OFF).

B3) Backend endpoint (add-only)
POST /api/v1/super/tenants/:tenantId/projects/:projectId/tasks/bulk
Body:
{
  rows: [
    { sectionName, taskTitle, description?, status?, priority?, dueDate?, assigneeEmails?, tags?, parentTaskTitle? }
  ],
  options?: {
    createMissingSections?: boolean,      // default true
    allowUnknownAssignees?: boolean,      // default false
    splitMultiAssigneeEstimate?: boolean  // ignore unless you already have estimates
  }
}
Returns:
{
  createdSections: number,
  createdTasks: number,
  createdSubtasks: number,
  skipped: number,
  errors: number,
  results: [
    { rowIndex, status: "created"|"skipped"|"error", reason?, sectionId?, taskId?, parentTaskId? }
  ]
}

Implementation notes:
- Must be per-row safe; one bad row does not block others.
- Must be tenant-scoped and super_user only.
- Must not create tasks for the wrong tenant/project.

B4) Section model compatibility
- If your app already has “task sections” for projects, reuse it.
- If your app DOES NOT have project task sections in schema:
  - Do NOT invent a brand new complex model here.
  - Instead, represent “sections” using an existing field (e.g., task.sectionName) if present.
  - If no existing field exists, add a minimal ADD-ONLY table:
    project_task_sections
    - id, tenantId, projectId, name, sortOrder, createdAt
  and add tasks.sectionId nullable FK
  But only do this if necessary and keep it minimal.
