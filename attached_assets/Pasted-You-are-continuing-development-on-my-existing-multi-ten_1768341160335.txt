FEATURE SET A: MULTI-ASSIGNEE TASKS & SUB-TASKS
===============================================================================
GOAL:
Allow Tasks and Sub-Tasks to be assigned to multiple users (2+ assignees),
while preserving backward compatibility with existing single-assignee behavior.

-------------------------------------------------------------------------------
A1) DATA MODEL (ADD-ONLY)
-------------------------------------------------------------------------------
Create join table:

task_assignees
- id (uuid)
- tenantId (uuid, indexed)
- taskId (uuid, FK tasks.id)
- userId (uuid, FK users.id)
- createdAt

Constraints:
- UNIQUE(taskId, userId)
- Users assigned MUST belong to the same tenant as the task.

Sub-Tasks:
- If sub-tasks are stored in the same tasks table (parentTaskId pattern),
  reuse task_assignees for both tasks and sub-tasks.

Backward compatibility:
- KEEP existing tasks.assignedToUserId column.
- DO NOT remove or repurpose it.

-------------------------------------------------------------------------------
A2) API BEHAVIOR (BACKWARD COMPATIBLE)
-------------------------------------------------------------------------------
Write:
- Accept new optional field:
  assigneeIds: string[]

Rules:
- If assigneeIds provided:
  - validate users exist and belong to same tenant
  - replace rows in task_assignees for that task
  - optionally set assignedToUserId = first assignee (legacy compatibility)
- If assigneeIds not provided:
  - preserve existing behavior using assignedToUserId

Read:
- Always return:
  assignees: [{ id, firstName, lastName, email }]
- If no task_assignees rows exist but assignedToUserId exists:
  - synthesize assignees = [assignedToUser]

Filtering:
- “My Tasks” logic must include tasks where:
  - assignedToUserId = currentUserId OR
  - currentUserId exists in task_assignees

A3) FRONTEND (TASK & SUB-TASK EDITORS)

A4) VALIDATION & SECURITY
